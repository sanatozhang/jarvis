# Coreguard 预测带报警机制重设 — 设计文档

- **日期**：2026-06-05
- **模块**：Coreguard 核心指标告警（`app/coreguard/`）
- **状态**：设计已确认 + **历史回验通过**，待写实现计划
- **作者**：sanato + Claude（brainstorming 协作）

> **2026-06-05 回验后修订**：原设计的"6 周同段 baseline"经实测发现 **RUM 数据仅保留 ~30 天**，
> 不可行。已改为 **方案 B：同 2h-of-day × 近 N 天密集 baseline**（详见 §2.2、§3）。回验结果见 §6.3。
> 关键结论：告警**量与旧机制相当**（旧 5.1/周 → 新 4.6/周 @k=3），但**构成显著变好**——
> 干掉噪声/方向 bug 类误报、换成有统计依据的告警，且**零逐指标调参**、方向 bug 结构性修复、真事故不漏。

## 1. 背景与问题

### 1.1 现状机制（"单小时 SHoW 对比"）

| 维度 | 现在做法 | 代码 |
|---|---|---|
| 颗粒度 | 每小时取**上一个完整自然小时** 1 个桶 `[H-1, H)`，对比**上周同 weekday 同小时** 1 个桶 | `demo_metric.py:66-77` |
| 阈值 | 固定百分比/百分点，逐指标人工调 | `metrics.yaml` 各 `threshold` |
| 防抖 | min_users 地板(300) + P1 连续 2 小时 breach + P0 立即报 | `metric_watcher.py:304-316` |

### 1.2 根因（两个表象，一条因果链）

用户提出的"颗粒度"和"阈值"两个问题，实为同一根因：

```
单小时桶样本小、方差天然大
  → SHoW 单点对比一比就抖（噪声大）
  → 为压误报只能把固定阈值越调越宽
  → 阈值宽到只剩"灾难级"才报（漏报真实小回归）
  → 偏离报警本意
```

`metrics.yaml` 里的调参痕迹即证据：成功率 `2.5→4.0pp`、tab 切换 `0.20→0.30`、内存 `0.25→0.10`（注释直言"25% 宽到几乎不动"）。

### 1.3 设计目标

在**保持小时级时效（1-3h 内发现）**的前提下，用统计方法从源头降噪，让阈值自动适配每个指标的固有波动，**彻底消灭逐指标手调阈值**。

## 2. 核心算法：预测带（Prediction Band）

每小时，对每个 `alert_enabled` 指标执行四步：

### 2.1 取当前值
当前**滚动 3h 窗口** `[now-3h, now)` 的真实值 `cur`。
（窗口放大到 3h 是免费降噪：一次 scalar 查 3h 与查 1h 都是 1 次请求，只是 start/end 拉宽；样本量 ×3 → 当前值和历史值都更稳。）

### 2.2 取历史同时段序列（方案 B：同时段 × 近 N 天密集）
取"**同 2h-of-day**"在**最近 N 天**（建议 N=14）各 1 个值 → `[v₁ … vₙ]`（offset = 1d/2d/…/14d，同一日内时段）。
这是"之前线的走势"——从"只比上周 1 个点"升级成"比近 14 天这个时段的整条分布"。

> **为什么不是原计划的"6 周同 weekday"**：实测 RUM 仅保留 ~30 天（§3.1），6 周（42d）取不到数。
> 30 天硬约束下，"强周季节性"和"稳带宽估计"二选一 → 选 B（同时段近 N 天），用 ~14 个点换稳定的带宽。
> 这批指标多是比率/百分位，对工作日不敏感，**日内周期**才是主因，B 恰好抓住它。
>
> 采用**季节性预期**（这个时段历史上通常是多少），**不是趋势外推**（流量驱动有强昼夜周期，外推会把"白天本该回升"误判为暴涨）。
> 代价：丢工作日/周末差异（如需保留可按 §8 的"工作日/周末分桶"扩展）。

### 2.3 算预测值 + 带宽（全靠数据，不手调）
```
μ = median(v₁ … vₙ)                       # 中位数，单次历史毛刺不污染预测
σ = max(1.4826 × MAD(v₁ … vₙ), floor)     # MAD 抗离群点；floor 防零宽带
上带 = μ + k·σ
下带 = μ − k·σ
```
- `k` 全局统一（**回验确定 k=3**，见 §6.3）
- `floor` 防零宽带：百分比类用绝对 pp（如 0.05pp）；latency/count 类用相对值（如 0.5%·μ）。
  metrics 源（memory/ANR/Hang）带很紧，floor 可能需单独上调（§6.3 待处理项）

### 2.4 判定
`cur` 穿带 → 异常（方向感知见 §4）；穿出 σ 数 = 严重度。

### 2.5 自适应证据（同一个 k=3，带宽自动不同）

| 指标 | 近 N 天同时段（示例取 6 点）| μ | 自动带宽(±3σ) | 解读 |
|---|---|---|---|---|
| crash_free_users（稳） | 99.7/99.6/99.8/99.5/99.7/99.6 | 99.65% | **±0.22pp** `[99.43, 99.87]` | 数据稳 → 带自动收到 ±0.2pp，掉 0.45pp 立刻抓到 |
| tab 切换耗时（抖） | 820/1180/950/1050/880/1120ms | 1000ms | **±530ms** `[470, 1534]` | 数据抖 → 带自动放到 ±53%，1300ms 不报，1600ms 才报 |

→ 整张 yaml 只剩一个 `k`，稳指标抓得细、抖指标不误报。

## 3. 数据来源与冷启动

**每次运行直接从 Datadog 在线回拉历史，不依赖本地快照表。**

snapshot 表才 11 天。Datadog scalar API 支持任意 start/end，每指标把原来 2 次查询扩成 `1 次 current + N 次历史同段`（同时段近 N 天，N=14）。

### 3.1 数据保留期（已实测，决定 baseline 用方案 B 的根因）
| 数据源 | 覆盖指标 | 实测保留期 |
|---|---|---|
| **RUM** | crash-free、各成功率、各延迟、FPS、卡顿、wifi 速度（**绝大多数告警指标**）| **~30 天**（28-31d 有数，32d 起返回空数组）|
| **metrics** | memory / ANR / Hang / Refresh（4 个）| **≥90 天**（87-90d 仍有数）|

→ 6 周（42d）baseline 对占多数的 RUM 指标取不到数据 → 必须用 §2.2 方案 B（近 N 天，N≤可用天数）。

### 3.2 成本
| 项 | 现在 | 新方案（N=14）|
|---|---|---|
| 每指标查询数 | 2 | ~15（或一次 timeseries 拉连续序列，离线切片）|
| 启用指标数 | ~16 | ~16 |
| 频率 | 每小时 1 次 | 每小时 1 次（不变）|

> 实测：Datadog timeseries API **无视请求的 interval，固定给 2h 分辨率且 ~361 点封顶**。
> 实现可二选一：(a) 每窗口用 scalar 精确查（量大但精确 3h）；(b) timeseries 分块拉 2h 序列离线切片（快但窗口 2h）。
> 串行 N 倍查询偏慢（`metric_watcher.py:292`）→ 改**小批量并发**（每批 5-8 个）。

### 3.3 三重兜底
1. **零冷启动、自愈**：不写回填脚本、不依赖快照表完整度。某天返回 null → 剔除该点，用剩余点算 median/MAD。
2. **数据不足兜底**：有效历史点 `< 3` → 不报警、只记录（复用 min_users gate 的"静默写快照"逻辑），避免冷启动误报风暴。
3. **零宽带兜底**：MAD=0（历史完全平）→ `σ = max(1.4826·MAD, floor)`，防任何微抖穿带。

### 3.4 快照表角色变更
从"预测数据源"降级为**审计日志 + 防抖依据**（查上一窗口是否 breach）。预测基准一律来自 Datadog 实时。

## 4. 方向感知判定（结构性消灭"方向配反"bug）

预测带是两侧的 `[下带, 上带]`，但**报不报警单侧、看方向**：

| 方向 | 例子 | 穿下带 | 穿上带 |
|---|---|---|---|
| down_is_bad（越高越好） | crash-free、各成功率、wifi 速度、FPS | 🔴 报警 | ✅ 静默（异常地好） |
| up_is_bad（越低越好） | 各延迟、ANR、内存、Hang、卡顿 | ✅ 静默 | 🔴 报警 |
| both | API 请求量 | 🔴 报警 | 🔴 报警 |

→ wifi 速度 +37.71% = 穿上带且 down_is_bad → **天然静默**，不会再误报。

### 4.1 从根上让 bug 不可能复发
wifi 事故根因：人在 yaml 手写 `up_is_bad`，而 Datadog widget 声明 `directionality: increase_better`，无人校验。

**结构性修复**：`dashboard_loader` 加载期以 Datadog widget 的 `comparison.directionality` 为真相源做交叉校验：
```
increase_better → down_is_bad
decrease_better → up_is_bad
neutral         → both
```
- 一致 → 通过
- **冲突 → 加载期 fail loud（报错/拒绝启用该指标），日志点名**

"方向写反"在启动期即被拦下，到不了发报警那一步。从"靠测试守"升级为"结构上不可能"。

## 5. 防抖、严重度、样本 gate

### 5.1 严重度：穿出几个 σ（连续异常分数）
| 穿出幅度 | 级别 |
|---|---|
| 3-4σ | 🟡 关注 |
| 4-6σ | 🟠 警告 |
| >6σ | 🔴 紧急 |

tier(P0/P1/P2) 不消失，角色变为**业务重要度/路由**，与 σ 幅度正交。卡片同时显示：`[P0] crash-free · 穿下带 6.1σ`。

### 5.2 防抖：统一 N=2 确认（已确认决策）
**所有穿带都要求上一个小时也穿带**，否则只记录不报。
- 取舍：即便强事故也等约 1 小时；但 3h 滑动窗 + 每小时评估 → 有效检测延迟约 2h，仍在 1-3h 容忍线内，真事故下一窗口必仍在。
- 实现复用 `_was_breached_in_prev_window`。
- σ 幅度只用于显示级别，不参与防抖。

### 5.3 样本 gate：保留 min_users
当前窗口 distinct user < min_users → 指标不可信 → 只记录不报。与 §3.2 的"历史有效点<3不报"构成数据可信度双闸。

### 5.4 每指标每小时判定流水线
```
1. 算带；历史有效点 < 3 → 只记录，跳过
2. 方向感知穿带？没穿 → healthy
3. min_users 不够 → 只记录，跳过
4. 算 σ 幅度 → 定级别（🟡/🟠/🔴）
5. 防抖：上一窗口也穿带才报，否则只记录
6. → 入飞书卡
```

### 6.3 回验结果（2026-06-05，18 个启用指标 × 30 天真实数据）

回验口径（受 Datadog 现实约束，结论偏保守）：2h 窗（设计 3h 更干净）、未套 min_users gate（高估告警量）、baseline=同时段近 14 天、防抖 N=2。

**① 选定 k=3：告警量与旧机制相当（不是砍半）：**

| k | 折合告警/周 | 有告警指标数 |
|---|---|---|
| 2.5 | 11.1 | 5 |
| **3.0（选定）** | **4.6** | 3 |
| 3.5 | 4.2 | 3 |
| 旧机制 | 5.1 | 5 |

> 注：早先版本误报"砍半（旧 10.7/周）"，是回验脚本把 `absolute_threshold` 指标（hang_rate）的旧逻辑
> 错当成相对 % 阈值、虚报 12 次所致。修正 `judge_old` 后旧机制实为 ~5.1/周，与新机制量相当。

**② 真正价值在告警"构成"变好（@k=3 逐指标 vs 旧）**：
- 干掉噪声/方向类误报：`wifi_speed` 1→0（方向 bug）、`highlight_render` 3→0、`home_file_list_load_p90` 1→0。
- 新增有据告警：`memory_usage` 0→2（+6% 内存涨）、`detail_tab_switch_p75` 4→6（含 59s glitch 离群）。
- 持平：`ai_task_transcribe_success_rate` 2=2（真事故，新旧都抓）。

**③ 三个不体现在"量"上的核心收益**：① **零逐指标调参**（整表只剩一个 k）；② 方向 bug **结构性修复**（wifi 1→0 实证）；③ 噪声告警换成**有统计依据**的告警（讲得清"为什么报"）。

**④ 真事故不漏**：转写两次掉点（88.1%/6.9σ、89.9%/5.4σ）新旧都抓到。

**回验暴露的实现待处理项**（见 §8）：tab_switch 噪声 + 59s glitch、N=2 吞单桶大尖峰、memory 带过紧、2 个指标数据问题（api_latency_p95 稀疏 / audio_upload_cloud_duration HTTP 400）；`absolute_threshold` 指标（hang_rate）新引擎沿用绝对红线、不套带。

## 7. 决策记录（brainstorming 已确认）

| 决策点 | 选定 |
|---|---|
| 检测时效 | 小时级（1-3h 内） |
| 带宽来源 | 数据自定（μ ± k·σ，不手调） |
| 预测基准 | **方案 B：同 2h-of-day × 近 N 天（N=14）中位数 + MAD**（原 6 周因 RUM 30 天保留期改） |
| **k 值** | **k=3（回验确定，§6.3）** |
| 窗口 × 频率 | 3h 滑动窗 + 每小时评估 |
| 方向判定 | 方向感知 + 加载期 directionality 交叉校验 |
| 防抖 | 统一 N=2 确认（所有穿带） |
| 严重度 | 穿出 σ 数（🟡3-4 / 🟠4-6 / 🔴>6），tier 作路由 |

## 8. 未决/后续（回验暴露，实现时处理）
- **tab_switch 噪声**：天生抖 + 出现 59297ms(59s) 的 2173σ glitch 离群点。靠 min_users gate(500) + 可选"current 值也做离群裁剪"压制。
- **N=2 吞单桶大尖峰**：highlight_render 22.5σ 孤立尖峰被防抖压掉——是"统一 N=2"的预期代价，已确认接受；如需可对"超大 σ（如 >10σ）"破例立即报。
- **memory_usage 带过紧**：太稳 → 带窄 → +6% 就报。给 metrics 源单独抬 floor。
- **数据问题指标**：`api_latency_p95` 有效点稀疏（n_eval=13）；`audio_upload_cloud_duration` 查询 HTTP 400（多公式 timeseries widget 需特殊处理）→ 这两个实现时单独修或先 `alert_enabled=false`。
- **工作日/周末分桶**（可选增强）：方案 B 当前丢工作日差异，如某指标周末差异大可分桶。
- **`both` 方向指标**（API 请求量）是否启用，后续按需。
