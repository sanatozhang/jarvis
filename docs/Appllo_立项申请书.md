## 1. 项目背景与痛点

### 1.1 现状

Plaud 硬件产品日活用户持续增长，用户在使用过程中遇到录音异常、同步失败、固件崩溃等技术问题后，会通过 Zendesk 工单系统联系 CSC 客服团队。客服同学在 Zendesk 内与用户沟通，当遇到无法独立判断的技术问题时，需要将工单转交研发工程师进行人工日志排查，才能给出答复。

#### 流程对比：优化前 vs 优化后

| 步骤 | ⏱️ 优化前（数小时 ～ 数天） | ⚡ 优化后（分钟级） |
|:---:|---|---|
| 1 | 👤 用户遇到技术问题，在 Zendesk 联系 CSC | 👤 用户遇到技术问题，在 Zendesk 联系 CSC |
| 2 | 💬 CSC 与用户沟通，**手动整理**聊天记录总结问题 | 💬 CSC 与用户沟通，判断是否需要技术排查 |
| 3 | 📋 CSC **跳转飞书**，手动创建技术工单，填写描述 | 🖱️ CSC **一键提交** Appllo，无需切换系统 |
| 4 | ⏳ **工单进入值班队列，等待工程师有空**（可能跨天） | 🤖 **AI 自动**下载日志、一次解密、二次脱密、匹配规则、分析根因 |
| 5 | 🔍 工程师**中断当前工作**，人工下载日志手动排查 | ✅ **< 5 分钟**输出根因分析 + 建议回复话术 |
| 6 | 📢 工程师将结论反馈给 CSC（再次等待） | 💬 CSC 确认后**直接在 Zendesk 回复用户** |
| 7 | 💬 CSC **再次联系**用户传达结果 | — 复杂工单自动创建 Linear Issue 通知工程师 |
| 8 | ✅ 问题关闭 | ✅ 问题关闭 |
| **总耗时** | **⛔ 数小时 ～ 数天** | **✅ 50% 工单 < 5 分钟** |
| **工程师占用** | **每日 2～4 小时，频繁打断** | **每日 < 30 分钟，专项处理** |
| **客户体验** | **长时间等待，易流失** | **同一次沟通内解决，满意度提升** |

**上线前后对比：**

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 工单处理周期 | 数小时 ～ 数天 | **50% 工单分钟级响应** |
| 工程师每日工单占用 | 2～4 小时 | **< 30 分钟** |
| 每周工单量（需工程师介入） | 50～70 个 | **目标 ~20 个** |
| CSC 独立处理率 | 0% | **目标 ≥ 60%** |
| 客户等待体验 | 长时间等待，体验差 | **实时响应，满意度提升** |

### 1.2 痛点

| 痛点 | 描述 |
|------|------|
| **工程师时间碎片化** | 工单随机分布在工作日，频繁打断研发节奏，严重影响深度工作 |
| **处理周期长** | 工程师不在线或排期满时，客户等待超过 24 小时，客诉升级风险高 |
| **客服依赖性强** | 客服无技术能力独立分析日志，所有技术问题必须排队等工程师 |
| **经验无法沉淀** | 相似问题反复排查，缺乏规则化、结构化的知识库 |
| **日志分析门槛高** | `.plaud` 专有格式日志需要解密和专业工具才能读取，非技术人员完全无法处理 |

### 1.3 迫切性

**用户侧：** 工单响应慢直接损害用户体验和品牌口碑。当前平均 1～3 工作日的等待周期，意味着用户在问题得到解决前可能已经流失或留下差评。随着 Plaud 产品出货量持续增长，若不解决响应速度问题，客诉升级和退款率的压力将同步放大。

**研发侧：** 工单排查是典型的"高打断、低创造"工作。工程师每天 2～4 小时被碎片化工单占用，严重压缩深度开发时间。工单量随用户规模线性增长，形成"增长陷阱"——业务越成功，研发团队被动支持负担越重，产品迭代速度反而下降。

**规模化拐点：** 当前每周 50～70 个工单已触及人工处理上限。不引入自动化，任何增长都将直接转化为工程师加班或 HC 需求，边际成本居高不下。

## 2. 目标与价值

### 2.1 核心目标

1. **极速响应，提升客户满意度**：将 50% 工单的处理时间从"1～3 工作日"压缩至"< 5 分钟"，用户在同一次沟通中即可获得答复，彻底消除等待体验
2. **客服赋能，独立处置技术问题**：CSC 客服同学无需等待工程师，在 Zendesk/Appllo 内一键触发分析，拿到根因结论和建议话术后直接回复用户
3. **工程师解放，专注核心研发**：将需要工程师介入的工单从每周 50～70 个降至 ~20 个，每日工单排查时间从 2～4 小时压缩至 30 分钟以内，研发精力回归产品迭代
4. **经验资产化，问题不重复消耗**：将排查经验沉淀为可持续迭代的规则库，相同问题 AI 直接命中，不再反复占用工程师时间

## 3. 技术方案概览

### 3.1 当前架构（MVP）

```
飞书多维表格 / Linear / 用户反馈表单
        ↓ Webhook / API 拉取
    Backend（FastAPI + SQLAlchemy + SQLite）
        ↓ 日志下载 → 一次解密 → 二次脱密 → 规则匹配 → 预提取
    AI Agent（Claude Code CLI / OpenAI Codex CLI）
        ↓ 结构化 JSON 结果
    Frontend（Next.js 15 + React 19 + Tailwind CSS 4）
        ↓ SSE 实时进度推送
    客服/工程师 Web 界面
```

| 组件 | 选型 | 理由 |
|------|------|------|
| Web 框架 | FastAPI | 异步高性能，适合 IO 密集型工单处理 |
| AI 引擎 | Claude Code CLI（主）+ Codex（备） | Claude 代码分析能力强，支持多路由策略 |
| 前端 | Next.js 15 App Router | SSE 实时推进度；中英双语 i18n 内置 |
| 工单来源 | 飞书 API + Linear Webhook | 覆盖现有两套工单流转系统 |
| 规则引擎 | Markdown + YAML frontmatter | 低门槛维护，工程师可直接 PR 更新规则 |
| 部署 | Docker Compose | 单机即可运行，迁移成本低 |
| **身份认证** | VPN + 手动配置用户名 | ⚠️ 当前方案，依赖内网访问，用户身份无法精确识别 |

**对比方案：**

| 方案 | 描述 | 放弃原因 |
|------|------|---------|
| 纯人工 RAG 问答 | 向量检索 + FAQ | 无法处理原始日志，需大量标注数据 |
| 接入 LLM API（无 Agent） | 直接调用 GPT/Claude API | 缺乏代码执行能力，无法动态分析日志结构 |
| **Appllo（当前方案）** | AI Agent 读取真实日志文件做分析 | 可执行代码、可读日志、可调用工具，分析深度最高 |

### 3.3 AI 分析能力演进路径

Appllo 的 AI 分析能力设计为三个阶段递进，**每个阶段的准确率都高于上一阶段**，且无需推倒重来：

| 维度 | 🛠️ Phase 1 · 规则驱动（当前） | 🔄 Phase 2 · 反馈闭环（近期） | 🧠 Phase 3 · 数据训练（中长期） |
|------|-------------------------------|-------------------------------|--------------------------------|
| **核心机制** | 工程师将排查经验整理为结构化规则库，规则引导 AI 定向分析 | CSC 对每次分析结果打分，反馈驱动规则持续优化 | 积累的标注样本用于 Prompt 微调与模型优化 |
| **数据来源** | 工程师经验（手动编写规则） | CSC 日常使用中的准确度评分 | 真实工单 + 日志 + CSC 标注的正确根因 |
| **维护方式** | 工程师 PR 更新规则，热重载生效 | 低质量规则自动标记，触发工程师复核 | 数据驱动自动优化，减少人工维护 |
| **准确率预期** | 高置信命中率目标 ≥ 70% | 命中率随反馈数据持续提升 | 突破 70%，对新问题类型具备泛化能力 |
| **最大优势** | 零标注成本，立即可用 | 标注无额外人力，随业务自然积累 | 准确率随规模自我进化，越用越准 |

**Phase 1 — 规则驱动（已上线）**

当前分析质量来源于工程师多年处理客诉积累的经验：工程师将排查思路、关键日志特征、典型根因整理为结构化规则（Markdown + YAML），由规则引擎在 AI 分析前进行关键词匹配和日志预提取，为 AI Agent 提供精准上下文。规则库可热重载，工程师发现新问题类型时即时补充，不需要发版。

**Phase 2 — 反馈闭环（近期规划）**

CSC 客服同学在使用 Appllo 分析结果时，可以对每次分析给出准确度反馈（"根因正确"/"根因有偏差"/"完全不对"）。这些反馈数据将：
- **短期**：自动标记低质量规则，触发工程师复核和优化，形成规则迭代闭环
- **中期**：积累带标注的"工单 + 日志 + 正确根因"样本对，为下一阶段奠基

**Phase 3 — 数据驱动训练（中长期）**

当反馈样本积累到足够规模后，可基于真实标注数据对模型进行微调或 Prompt 优化，使 AI 的分析视角越来越贴近 Plaud 产品的实际问题模式，逐步实现：
- 高置信命中率从当前目标 70% 持续提升
- 降低对规则人工维护的依赖
- 对新问题类型具备一定的泛化能力

> 这一演进路径的核心优势在于：**数据来自真实工单处理过程，标注由最了解产品的 CSC 和工程师完成，无需额外的标注人力投入**，准确率随业务运转自然提升。

### 3.4 可行性论证

- **技术可行性**：MVP 已上线，核心链路（飞书拉取→日志下载→一次解密→二次脱密→AI 分析→结果回写）均已验证
- **模型能力**：Claude Code 具备文件读取、代码执行、日志分析的 Agentic 能力，天然适配本场景
- **规则系统**：已建立热重载规则引擎，无需重启即可更新分析规则，运维成本极低
- **自我进化**：反馈闭环设计让系统准确率随使用量自然提升，无需额外数据采购或专项标注
- **扩展性**：Agent 路由支持多模型，可按问题类型切换 Claude/Codex/未来微调模型

**飞书机器人（Chatbot）**
- 客服在飞书群内 `@Appllo` 直接发起工单分析
- 机器人实时推送分析进度和结果（富文本卡片）
- 支持指令：`/analyze <工单ID>`、`/status <任务ID>`、`/report week`
- 工程师可在飞书内直接审核 AI 结论并一键确认或驳回

**飞书多维表格双向同步**
- 分析结果自动回填至对应行（问题类型、根因、置信度、建议回复）
- 工单状态变更（已解决/转人工）实时同步至多维表格字段
- 支持客服在多维表格内修改字段后触发重新分析

**飞书审批流**
- `needs_engineer: true` 的工单自动发起飞书审批，指定 oncall 工程师
- 工程师审批通过后自动关联 Linear Issue

**技术实现：**

```
飞书事件订阅（Webhook）
  → /api/feishu/event
  → EventDispatcher（按 event_type 路由）
      ├── im.message.receive_v1  → 机器人指令处理
      ├── bitable.record.updated → 多维表格变更触发重分析
      └── approval.instance.cc   → 审批结果回调

飞书消息推送
  → FeishuNotifier（封装卡片消息模板）
  → 分析开始 / 进度更新 / 完成 / 需人工介入 四类卡片
```

新增模块：`feishu_bot.py`、`feishu_notifier.py`、`routers/feishu_event.py`

### 5.2 Zendesk 深度集成

> **对处理时效的提升：** Zendesk 是 CSC 的主战场，深度集成后 CSC 无需切换系统——工单在 Zendesk 内自动触发分析，结果直接写回为回复草稿，客服一键发送即可。从"发现问题 → 拿到答案 → 回复用户"的全链路时间从数小时压缩至分钟级，客户在单次沟通内即可获得完整答复。

**集成架构：**

```
Zendesk Ticket
    ↓ Trigger / Webhook（工单创建/更新）
/api/zendesk/webhook
    ↓
Appllo 分析 Pipeline
    ↓ 分析完成
Zendesk API
    ├── 添加内部备注（根因 + key_evidence）
    ├── 添加用户可见回复草稿（user_reply）
    ├── 更新自定义字段（问题类型、置信度、AI处理状态）
    └── 变更工单 Assignee（needs_engineer → 转指定组）
```

**目标集成能力：**

**工单自动入队**
- Zendesk Trigger：工单标签含 `technical` 或 Tag 匹配规则时，自动 POST 至 Appllo Webhook
- 支持附件（日志文件）自动下载并送入分析 Pipeline
- 工单去重：同一 Zendesk Ticket ID 30 分钟内不重复触发

**分析结果写回**

| Appllo 字段 | Zendesk 自定义字段 |
|-------------|-------------------|
| `problem_type` | `appllo_problem_type` |
| `confidence` | `appllo_confidence` |
| `needs_engineer` | `appllo_needs_engineer` |
| `task_id` | `appllo_task_id`（可跳转 Appllo 详情页） |

**工单路由**
- `confidence: high` + `needs_engineer: false` → 自动分配至"客服处理"组
- `confidence: low` 或 `needs_engineer: true` → 升级至"技术支持"组
- 分析失败 → 保持默认分配，添加内部备注说明原因

**Zendesk 侧边栏 App**
- 在工单详情页侧边栏展示 Appllo 分析结果
- 支持客服直接在 Zendesk 内触发重新分析
- 展示历史分析记录和处理时间线

新增模块：`routers/zendesk.py`、`services/zendesk_client.py`、`services/zendesk_mapper.py`、`zendesk-app/`（侧边栏 App）

### 5.4 Google SSO 登录集成

> **为什么要做：** 当前系统依赖 VPN 接入内网 + 手动配置用户名，存在两个核心问题：一是访问门槛高，非内网环境需要手动连 VPN；二是用户身份不可信，手动填写的用户名无法与真实员工身份绑定，操作日志无法溯源。迁移至 Google SSO 后，既消除 VPN 依赖，又实现精确的员工身份识别，同时也是 Phase 3 多团队扩展的必要前提。

**现状 vs 目标对比：**

| 维度 | 当前（VPN + 手动用户名） | 目标（Google SSO） |
|------|------------------------|--------------------|
| **访问方式** | 需连接公司 VPN，外网无法直接访问 | 浏览器直接访问，Google 账号一键登录 |
| **身份识别** | 手动填写用户名，无法验证真实性 | 与 Google Workspace 账号绑定，精确识别员工 |
| **权限管理** | 无细粒度权限，所有人看到相同界面 | 按角色（CSC / 工程师 / 管理员）控制页面和操作权限 |
| **操作审计** | 操作日志无法与真实用户关联 | 所有操作记录绑定 Google 账号，完整可溯源 |
| **安全性** | VPN 密钥泄露即可访问全系统 | OAuth 2.0 标准，Google 负责凭证安全 |
| **多团队支持** | 无法区分不同团队成员 | 可按 Google 群组自动分配团队和权限 |

**技术实现：**

- 使用 Google OAuth 2.0 / OpenID Connect，后端验证 `id_token` 后签发 JWT
- 用户首次登录自动创建账号，邮箱域名限制为公司域（`@plaud.ai` 等）
- 角色映射：Google 群组 → Appllo 角色（CSC 组 → 客服角色，Engineering 组 → 工程师角色）
- 现有手动用户名账号提供一次性迁移工具，历史操作记录保留

新增模块：`routers/auth.py`、`services/google_oauth.py`、`middleware/auth_middleware.py`

## 6. 云端部署方案

### 6.1 云服务选型

| 组件 | 选型 | 备注 |
|------|------|------|
| 容器编排 | **AWS ECS** / **EKS** | 支持自动扩缩容，与其他 AWS 服务深度集成 |
| 前端托管 | **Vercel** | Next.js 原生支持，全球 CDN |
| 数据库 | **AWS RDS PostgreSQL** | 替换 SQLite，多可用区高可用 |
| 任务队列 | **AWS ElastiCache Redis + ARQ** | 与现有 AsyncIO 架构兼容，托管免运维 |
| 日志/缓存存储 | **AWS S3** | 替换本地 workspaces 目录，按量计费 |
| 密钥管理 | **AWS Secrets Manager** | API Key 统一托管，自动轮换 |
| 监控告警 | **AWS CloudWatch** + **Datadog** | 接口延迟、队列积压、AI 命中率 |
| 日志收集 | **AWS CloudWatch Logs** | 结构化日志检索，与 ECS 原生集成 |

### 6.2 高可用设计

| 维度 | 方案 |
|------|------|
| **Backend 无状态化** | 工作目录迁移至 AWS S3，Backend 实例可任意替换 |
| **任务持久化** | 任务状态持久化至 PostgreSQL，Worker 崩溃后自动恢复 |
| **Worker 自动扩容** | 队列积压超阈值时自动扩容 Worker 实例（HPA） |
| **多可用区部署** | RDS / Redis 跨 AZ 主从，单 AZ 故障自动切换 |
| **健康检查** | 现有 `/api/health` 接口接入负载均衡探针 |
| **Claude API 限流** | 本地令牌桶限流 + 指数退避重试，防止 429 级联失败 |

### 6.3 数据迁移计划

```
阶段1：数据库迁移
  SQLite → PostgreSQL
  - Alembic 迁移脚本
  - 双写验证（1周）→ 切换读流量 → 停写 SQLite

阶段2：存储迁移
  本地 workspaces/ → AWS S3
  - 修改 WorkspaceManager 支持 S3 驱动
  - 保留本地驱动作为开发环境 fallback

阶段3：队列迁移
  内存任务队列 → AWS ElastiCache Redis + ARQ
  - ARQ 与现有 AsyncIO worker 架构兼容
  - 任务状态模型无需改动

阶段4：容器化迁移
  Docker Compose 单机 → AWS ECS / EKS
  - 现有 Dockerfile 直接复用
  - 补充 ECS Task Definition / Service 配置
```
