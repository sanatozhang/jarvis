# Appllo 技术项目立项申请书

**项目名称：** Appllo —— AI 驱动的客诉工单智能分析平台
**项目发起人：** sanato
**申请日期：** 2026-03-17
**项目状态：** MVP 已上线运行，申请正式立项以获取持续资源投入

---

## 1. 项目背景与痛点

### 1.1 现状

PLAUD 硬件产品日活用户持续增长，用户在使用过程中遇到录音异常、同步失败、固件崩溃等技术问题后，会通过 Zendesk 工单系统联系 CSC 客服团队。客服同学在 Zendesk 内与用户沟通，当遇到无法独立判断的技术问题时，需要将工单转交研发工程师进行人工日志排查，才能给出答复。

**上线前数据：**
- 每周技术类工单量：**50～70 个**
- 工单处理周期：**平均 1～3 个工作日**（取决于工程师当前排期）
- 工程师每日投入工单排查时间：**约 2～4 小时**（跨多名工程师）
- 客服同学完全依赖工程师响应，无法独立处置技术问题

**上线后数据（当前）：**
- 系统每日自动处理工单：**约 30 个**
- AI 分析平均耗时：**< 5 分钟**（分钟级响应）
- 需人工介入比例：显著降低

### 1.2 痛点

| 痛点 | 描述 |
|------|------|
| **工程师时间碎片化** | 工单随机分布在工作日，频繁打断研发节奏，严重影响深度工作 |
| **处理周期长** | 工程师不在线或排期满时，客户等待超过 24 小时，客诉升级风险高 |
| **客服依赖性强** | 客服无技术能力独立分析日志，所有技术问题必须排队等工程师 |
| **经验无法沉淀** | 相似问题反复排查，缺乏规则化、结构化的知识库 |
| **日志分析门槛高** | `.plaud` 专有格式日志需要解密和专业工具才能读取，非技术人员完全无法处理 |

### 1.3 迫切性

随着 PLAUD 产品出货量增加，工单量预计将持续线性增长。如不建立自动化分析能力，工程师时间成本将随用户规模等比扩张，形成"增长陷阱"——业务越成功，技术团队被动支持负担越重，研发效率反而下降。

## 2. 目标与价值

### 2.1 核心目标

1. **工程师解放**：将技术类工单从"人工必须处理"转变为"AI 优先，工程师兜底"
2. **客服赋能**：让客服同学能独立在平台上发起分析、跟踪进展、回复用户，无需等待工程师
3. **经验资产化**：将排查经验沉淀为可维护的规则库，持续迭代提升命中率
4. **响应时效**：将工单平均处理时间从"1～3 工作日"压缩至"分钟级"

### 2.2 量化指标

**技术指标：**

| 指标 | 目标值 |
|------|--------|
| AI 分析平均耗时 | ≤ 5 分钟 |
| 系统可用性 | ≥ 99.5% |
| AI 分析置信度（high）命中率 | ≥ 70% |
| 规则库覆盖问题类型数 | ≥ 30 类 |
| 支持并发分析任务数 | ≥ 10 |

**业务价值：**

| 指标 | 上线前 | 当前/目标 | 提升幅度 |
|------|--------|-----------|---------|
| 周工单工程师介入数量 | 50～70 个 | 目标 ~20 个 | 降低 50%+ |
| 工单平均响应时间 | 1～3 工作日 | < 5 分钟 | 提升 100x |
| 工程师每日工单处理时长 | 2～4 小时 | < 30 分钟 | 节省 85%+ |
| 客服独立处理率 | 0% | 目标 ≥ 60% | 从零到有 |

**折算人力成本（保守估算）：**
- 节省工程师时间：~2h/天 × 250 工作日 = **500 人时/年**
- 按工程师时薪折算，相当于节省约 **0.25 个工程师 HC/年**

## 3. 技术方案概览

### 3.1 当前架构（MVP）

```
飞书多维表格 / Linear / 用户反馈表单
        ↓ Webhook / API 拉取
    Backend（FastAPI + SQLAlchemy + SQLite）
        ↓ 日志下载 → 解密 → 规则匹配 → 预提取
    AI Agent（Claude Code CLI / OpenAI Codex CLI）
        ↓ 结构化 JSON 结果
    Frontend（Next.js 15 + React 19 + Tailwind CSS 4）
        ↓ SSE 实时进度推送
    客服/工程师 Web 界面
```

| 组件 | 选型 | 理由 |
|------|------|------|
| Web 框架 | FastAPI | 异步高性能，适合 IO 密集型工单处理 |
| AI 引擎 | Claude Code CLI（主）+ Codex（备） | Claude 代码分析能力强，支持多路由策略 |
| 前端 | Next.js 15 App Router | SSE 实时推进度；中英双语 i18n 内置 |
| 工单来源 | 飞书 API + Linear Webhook | 覆盖现有两套工单流转系统 |
| 规则引擎 | Markdown + YAML frontmatter | 低门槛维护，工程师可直接 PR 更新规则 |
| 部署 | Docker Compose | 单机即可运行，迁移成本低 |

**对比方案：**

| 方案 | 描述 | 放弃原因 |
|------|------|---------|
| 纯人工 RAG 问答 | 向量检索 + FAQ | 无法处理原始日志，需大量标注数据 |
| 接入 LLM API（无 Agent） | 直接调用 GPT/Claude API | 缺乏代码执行能力，无法动态分析日志结构 |
| **Appllo（当前方案）** | AI Agent 读取真实日志文件做分析 | 可执行代码、可读日志、可调用工具，分析深度最高 |

### 3.3 可行性论证

- **技术可行性**：MVP 已上线，核心链路（飞书拉取→日志解密→AI 分析→结果回写）均已验证

- **扩展性**：Agent 路由支持多模型，可按问题类型切换 Claude/Codex/未来自研模型

## 5. Phase 2 深度集成方案

### 5.1 飞书深度集成

**现有能力盘点：**

| 能力 | 现状 |
|------|------|
| 从飞书多维表格拉取工单 | ✅ 已实现 |
| 下载飞书附件（日志文件） | ✅ 已实现 |
| 分析结果回写飞书 | ✅ 已实现（一键转工单） |
| 飞书消息推送通知 | ❌ 未实现 |
| 飞书机器人交互 | ❌ 未实现 |
| 飞书审批流集成 | ❌ 未实现 |

**新增目标能力：**

**飞书机器人（Chatbot）**
- 客服在飞书群内 `@Appllo` 直接发起工单分析
- 机器人实时推送分析进度和结果（富文本卡片）
- 支持指令：`/analyze <工单ID>`、`/status <任务ID>`、`/report week`
- 工程师可在飞书内直接审核 AI 结论并一键确认或驳回

**飞书多维表格双向同步**
- 分析结果自动回填至对应行（问题类型、根因、置信度、建议回复）
- 工单状态变更（已解决/转人工）实时同步至多维表格字段
- 支持客服在多维表格内修改字段后触发重新分析

**飞书审批流**
- `needs_engineer: true` 的工单自动发起飞书审批，指定 oncall 工程师
- 工程师审批通过后自动关联 Linear Issue

**技术实现：**

```
飞书事件订阅（Webhook）
  → /api/feishu/event
  → EventDispatcher（按 event_type 路由）
      ├── im.message.receive_v1  → 机器人指令处理
      ├── bitable.record.updated → 多维表格变更触发重分析
      └── approval.instance.cc   → 审批结果回调

飞书消息推送
  → FeishuNotifier（封装卡片消息模板）
  → 分析开始 / 进度更新 / 完成 / 需人工介入 四类卡片
```

新增模块：`feishu_bot.py`、`feishu_notifier.py`、`routers/feishu_event.py`

### 5.2 Zendesk 深度集成

**集成架构：**

```
Zendesk Ticket
    ↓ Trigger / Webhook（工单创建/更新）
/api/zendesk/webhook
    ↓
Appllo 分析 Pipeline
    ↓ 分析完成
Zendesk API
    ├── 添加内部备注（根因 + key_evidence）
    ├── 添加用户可见回复草稿（user_reply）
    ├── 更新自定义字段（问题类型、置信度、AI处理状态）
    └── 变更工单 Assignee（needs_engineer → 转指定组）
```

**目标集成能力：**

**工单自动入队**
- Zendesk Trigger：工单标签含 `technical` 或 Tag 匹配规则时，自动 POST 至 Appllo Webhook
- 支持附件（日志文件）自动下载并送入分析 Pipeline
- 工单去重：同一 Zendesk Ticket ID 30 分钟内不重复触发

**分析结果写回**

| Appllo 字段 | Zendesk 自定义字段 |
|-------------|-------------------|
| `problem_type` | `appllo_problem_type` |
| `confidence` | `appllo_confidence` |
| `needs_engineer` | `appllo_needs_engineer` |
| `task_id` | `appllo_task_id`（可跳转 Appllo 详情页） |

**工单路由**
- `confidence: high` + `needs_engineer: false` → 自动分配至"客服处理"组
- `confidence: low` 或 `needs_engineer: true` → 升级至"技术支持"组
- 分析失败 → 保持默认分配，添加内部备注说明原因

**Zendesk 侧边栏 App**
- 在工单详情页侧边栏展示 Appllo 分析结果
- 支持客服直接在 Zendesk 内触发重新分析
- 展示历史分析记录和处理时间线

新增模块：`routers/zendesk.py`、`services/zendesk_client.py`、`services/zendesk_mapper.py`、`zendesk-app/`（侧边栏 App）

## 6. 云端部署方案

### 6.1 云服务选型

| 组件 | 选型 | 备注 |
|------|------|------|
| 容器编排 | **阿里云 ACK** 或 **AWS ECS** | 支持自动扩缩容 |
| 前端托管 | **Vercel** | Next.js 原生支持，全球 CDN |
| 数据库 | **阿里云 RDS PostgreSQL** | 替换 SQLite，主从高可用 |
| 任务队列 | **Redis + ARQ** | 与现有 AsyncIO 架构兼容 |
| 日志/缓存存储 | **阿里云 OSS** | 替换本地 workspaces 目录 |
| 密钥管理 | **阿里云 KMS** | API Key 统一托管 |
| 监控告警 | **Datadog** 或 **阿里云 ARMS** | 接口延迟、队列积压、AI 命中率 |
| 日志收集 | **阿里云 SLS** | 结构化日志检索 |

### 6.2 高可用设计

| 维度 | 方案 |
|------|------|
| **Backend 无状态化** | 工作目录迁移至 OSS，Backend 实例可任意替换 |
| **任务持久化** | 任务状态持久化至 PostgreSQL，Worker 崩溃后自动恢复 |
| **Worker 自动扩容** | 队列积压超阈值时自动扩容 Worker 实例（HPA） |
| **多可用区部署** | RDS / Redis 跨 AZ 主从，单 AZ 故障自动切换 |
| **健康检查** | 现有 `/api/health` 接口接入负载均衡探针 |
| **Claude API 限流** | 本地令牌桶限流 + 指数退避重试，防止 429 级联失败 |

### 6.3 数据迁移计划

```
阶段1：数据库迁移
  SQLite → PostgreSQL
  - Alembic 迁移脚本
  - 双写验证（1周）→ 切换读流量 → 停写 SQLite

阶段2：存储迁移
  本地 workspaces/ → OSS
  - 修改 WorkspaceManager 支持 OSS 驱动
  - 保留本地驱动作为开发环境 fallback

阶段3：队列迁移
  内存任务队列 → Redis + ARQ
  - ARQ 与现有 AsyncIO worker 架构兼容
  - 任务状态模型无需改动

阶段4：容器化迁移
  Docker Compose 单机 → ACK/ECS
  - 现有 Dockerfile 直接复用
  - 补充 K8s Deployment / Service YAML
```

## 8. 风险评估

### 8.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| Claude API 响应慢/不稳定 | 中 | 中 | 已实现 Codex 备用路由；超时自动重试 |
| 日志格式升级导致解密失败 | 低 | 高 | 解密模块已独立封装，升级只需修改 `services/decrypt.py` |
| AI 分析结果误判 | 中 | 中 | `confidence` 字段低置信度自动转人工；工程师 oncall 兜底 |
| 并发工单量激增导致队列积压 | 低 | 中 | 任务队列已实现，Phase 2 引入 Worker HPA 自动扩容 |
| Zendesk API 限流（700 req/min） | 中 | 中 | 本地队列批量写回，削峰填谷 |
| 飞书 Webhook 事件重复投递 | 高 | 低 | 幂等处理（event_id 去重） |

### 8.2 资源风险

| 风险 | 应对措施 |
|------|---------|
| Anthropic API 涨价 | 规则引擎 L1 预提取可降低 token 消耗；支持切换 Codex 等替代模型 |
| 核心维护工程师离职 | CLAUDE.md 已详细记录架构、部署、已知问题，知识显式沉淀 |
| 云服务成本超预期 | OSS 日志文件 TTL 自动清理；Worker 按需弹性伸缩 |

### 8.3 兼容风险

| 风险 | 应对措施 |
|------|---------|
| 飞书 API 接口变更 | 飞书集成已模块化，变更只影响 `services/feishu.py` |
| Zendesk / Linear API 变更 | 各封装独立 client 层，变更只影响单文件 |
| 新设备/新固件日志结构变化 | 规则引擎支持热重载，新规则 PR 合并后立即生效，无需发版 |

### 8.4 合规风险

| 风险 | 应对措施 |
|------|---------|
| 用户日志数据出境（送往 Claude API） | 日志仅含设备状态码和时间戳，不含用户 PII；可评估私有化部署方案 |
| 日志留存合规 | 分析完成后工作区自动清理，不长期存储用户设备日志 |
